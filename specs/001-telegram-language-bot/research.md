# Research: планировщик и рантайм (Phase 0)

**Дата**: 2026-04-22 | Связано с: [plan.md](./plan.md)

## Требование

Фоновые задачи (напоминания, служебные тики) в **одном процессе** с FastAPI + aiogram, **без** отдельных worker-процессов (Celery, RQ, отдельный cron-контейнер).

## Варианты

| Подход | Плюсы | Минусы |
|--------|--------|--------|
| **APScheduler 4.x + AsyncIOScheduler** | Зрелая модель cron/interval, интеграция с asyncio, один loop | Дополнительная зависимость |
| **Чистый asyncio**: `asyncio.create_task` + `while` + `sleep` | Нет лишних библиотек | Ручная аккуратность с отменой, меньше выразительности для cron |
| **arq / dramatiq** | Надёжные очереди | Отдельный worker — **не подходит** под ограничение |

## Решение

Использовать **APScheduler 4.x** с **`AsyncIOScheduler`**, старт/стоп в **FastAPI lifespan**, все job-корутины вызывают те же **domain-сервисы**, что и бот.

**Альтернатива** (если захотите ноль зависимостей): две корутины `reminder_tick()` и `housekeeping_tick()` на `asyncio.sleep` — допустимо при простом MVP; в плане зафиксирован APScheduler как основной для читаемости расписаний.

## uvicorn + aiogram

- Один **event loop**: aiogram 3 работает на asyncio; webhook-обработчик FastAPI передаёт апдейт в dispatcher через `feed_update` / совместимый паттерн документации aiogram для вебхуков.
- **Конкурентность**: по умолчанию один воркер; при необходимости масштабирования — отдельное архитектурное решение (вне MVP).

## Redis в одном процессе

- **Dedup** `update_id`: `SET key NX EX ttl` — защита от повторной доставки webhook.
- **Напоминания**: флаг «уже отправили за локальный день» с TTL до конца суток по UTC или логика в БД — деталь в data-model / реализации.
